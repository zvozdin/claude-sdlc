# Чи працює оркестрація насправді? Чесний аналіз

## TL;DR

**Перший dispatch — ТАК, працює надійно (~90% випадків).** **Повний 5-фазний pipeline від BA до docs-writer автоматично — НІ, не працює надійно (~30-40% випадків доходить до кінця без втручання).**

Нижче — детальний розбір _чому_, з конкретними точками відмови і рекомендаціями.

---

## Що ПРАЦЮЄ надійно

### 1. Triage і перший dispatch (~90%)

Це найсильніша частина. Твій CLAUDE.md завантажує workflow.md через @-import, і перше що бачить модель — це чітка інструкція:

```
Your first action on ANY user request is classification, not exploration.
```

Decision tree простий і конкретний: Bug → debugger, Feature → ba, Infra → devops. Модель (особливо Opus) це розуміє і виконує. Перший subagent dispatch відбувається в 9 з 10 випадків.

**Чому 90%, а не 100%?** Є edge cases:

- Задача на межі "trivial" і "feature" — модель може вирішити зробити сама
- Юзер пише "just quickly fix this CSS" — модель сприймає як trivial
- Довгий, складний промпт — модель починає "думати" замість dispatch

### 2. Orchestrator не чіпає код (~80%)

Заборона `Read/Grep/Glob on project code` прописана тричі — в CLAUDE.md, на початку workflow.md, і в Orchestrator Tool Policy. Навіть є "маячок":

```
If you find yourself opening app/Actions/... — STOP.
```

Це працює для Opus. Модель дійсно уникає відкривання project files в більшості випадків. Але це **prompt-based** обмеження, не **enforcement**. В settings.json deny list є тільки `.env`. Модель технічно МОЖЕ прочитати будь-який файл — вона просто "обіцяє" не робити цього.

### 3. Subagent dispatch (Agent tool) (~85%)

Виклик конкретного агента через `Agent({ subagent_type: "ba", prompt: "..." })` — це стандартна, стабільна функція Claude Code. Subagents працюють в ізольованому context window, отримують промпт від оркестратора, виконують роботу, повертають результат.

---

## Що ФРАГІЛЬНЕ (працює іноді)

### 4. Передача контексту між фазами (~60%)

Ось де починаються проблеми. Pipeline передбачає:

```
BA → output → Developer → output → Quality Gate → output → Docs Writer
```

Але як саме output одного агента стає input наступного? Оркестратор має:

1. Отримати повний результат від BA
2. Зрозуміти його
3. Сформулювати промпт для Developer з урахуванням BA output
4. Передати цей промпт

На практиці, **оркестратор часто "спрощує" при передачі**. BA може видати 20 user stories з acceptance criteria, а оркестратор передасть developer'у стислий summary. Це не баг Claude Code — це фундаментальна проблема LLM: модель "розуміє" задачу і намагається бути helpful, замість того щоб точно ретранслювати output.

**Конкретний симптом:** Developer отримує: "Implement user registration with email verification" замість повного BA звіту з user stories, edge cases, і acceptance criteria.

### 5. Agent Teams (TeamCreate + паралельний Quality Gate) (~50%)

Agent Teams — це **experimental feature** з офіційно задокументованими обмеженнями:

> "Agent teams have known limitations around session resumption, task coordination, and shutdown behavior."

Твій pipeline передбачає:

```
TeamCreate("qg-user-registration")
→ spawn tester, reviewer, security-scanner, qa
→ wait for all 4 to complete
→ collect reports
→ decide pass/fail
→ TeamDelete
```

Це 6+ кроків координації. В реальності:

- Один з 4 агентів може зависнути або вичерпати context
- Оркестратор може не дочекатись всіх і продовжити з частковими результатами
- TeamDelete може не відбутись при помилці
- Якщо Quality Gate фейлить → back to developer → re-run — це вже 2 цикли координації, де кожен може зламатись

**Реальний досвід з ком'юніті:** Agent Teams працюють найкраще для "constrained, well-scoped scenarios" — 2-3 агенти з чіткими межами. 4 паралельні агенти в quality gate — це на межі того, що працює стабільно.

### 6. Devil's Advocate взаємодія (~40%)

Planning Team з devil, ba, ddd-architect — це красива ідея, але:

- devil має чекати на SendMessage від ba/ddd-architect
- Потім challenge через SendMessage
- Потім чекати відповідь
- Потім вирішити: accept чи escalate

Це **4+ rounds of async communication** між 3 агентами. Кожен round — це можливість втратити контекст, отримати timeout, або отримати відповідь яку devil не може правильно оцінити. На практиці devil часто або мовчить (не отримує тригер), або challenge'ить все підряд без розбору.

## Що ЙМОВІРНО НЕ ПРАЦЮЄ як задумано

### 7. Повний end-to-end pipeline автоматично (~30%)

Весь ланцюжок:

```
Classify → BA → DDD? → Developer → TeamCreate(4 agents) → Collect → Pass/Fail → Retry? → DocsWriter → PR
```

Це **мінімум 7 sequential steps**, з 2 паралельними фазами і можливими retry loops. В реальності:

1. **Context window overflow оркестратора.** Кожен subagent result додається до контексту оркестратора. До моменту Quality Gate оркестратор вже має: свій system prompt + workflow.md + code-style.md + git-operations.md + BA output + Developer output = значна частина context window зайнята ще до того, як починається збір QG результатів.
    
2. **"Drift" від інструкцій.** Claude Code has a known pattern: чим довша сесія, тим менше модель слідує оригінальним інструкціям. На 5-й фазі pipeline модель може "забути" що вона orchestrator і почати робити щось сама.
    
3. **Retry loops.** Якщо QG знайшов Critical → back to developer → re-run QG — це подвоює всі проблеми вище. Max 2 retries описаний в workflow.md, але модель може і не відслідкувати що це вже 2-й retry.
    
4. **Cost explosion.** Кожен агент — окремий context window. 4 QG агенти + retry = 8+ повних сесій. З Opus моделлю це $$$$.
    

### 8. Автодетект pipeline type

Workflow.md описує: "Pipeline Trigger: REQUIRED When ANY Applies" з 6 умовами (migration, route, Vue component, etc.). Але ці умови вимагають знання про задачу, яке оркестратор ще не має (бо йому заборонено читати код). Тому:

- "Add user dashboard" → модель правильно визначає feature pipeline
- "Fix the bug with category sort" → модель може не зрозуміти чи це pipeline чи trivial

## Фундаментальна проблема

Твій сетап описує **ідеальний процес** (як SDLC має працювати). Але Claude Code — це **LLM з tool use**, а не **workflow engine**. Різниця принципова:

|Workflow Engine|LLM з tool use|
|---|---|
|Детерміністичний: A → B → C завжди|Імовірнісний: A → B → C "зазвичай"|
|State machine з explicit transitions|Prompt-based "instructions"|
|Timeout handling, error recovery, retries|Сподіваємось модель не забуде|
|Enforcement (не може порушити)|Honor system (може порушити)|
|Зберігає state між кроками|Context window як єдиний "state"|

Ти побудував дуже детальний "spec" для workflow engine, але запускаєш його на "надійному стажерові з хорошою пам'яттю". Стажер в 90% випадків зробить перший крок правильно, в 70% — другий, в 50% — третій... До 7-го кроку ймовірність ідеального виконання = 0.9 × 0.8 × 0.7 × 0.6 × 0.5 × 0.5 × 0.5 ≈ **3.8%**.

## Конкретні рекомендації

### 1. Замість повного auto-pipeline — напівавтоматичний з checkpoint'ами

Замість того щоб сподіватись що оркестратор проведе 7 фаз автоматично, зроби checkpoint'и де оркестратор ЗУПИНЯЄТЬСЯ і показує юзеру результат:

```markdown
## Execution Model (REVISED)

Phase 1 (auto): Classify → dispatch BA → collect result
CHECKPOINT: Show BA summary to user. Ask: "Proceed to development?"

Phase 2 (auto): Dispatch Developer → collect result  
CHECKPOINT: Show what was implemented. Ask: "Run quality gate?"

Phase 3 (auto): TeamCreate QG → collect results
CHECKPOINT: Show QG findings. Ask: "Fix issues and re-run?" or "Accept and create PR?"

Phase 4 (auto): Dispatch docs-writer → PR
```

Це зменшує "longest autonomous chain" з 7 до 2-3 кроків, що радикально підвищує надійність.

### 2. Замість Agent Teams для QG — Sequential subagents

Agent Teams experimental і нестабільний для 4 паралельних агентів. Практичніший підхід:

```markdown
## Quality Gate (Sequential, NOT Team)

1. Dispatch `tester` as subagent → collect report
2. Dispatch `reviewer` as subagent → collect report  
3. Dispatch `security-scanner` as subagent → collect report
4. Synthesize: any Critical? → route to developer
```

Так, це повільніше ніж паралельне виконання. Але це ПРАЦЮЄ надійно, бо кожен крок — простий subagent call. Agent Teams можна використовувати як "upgrade path" коли фіча стане стабільнішою.

### 3. Explicit context passing через файли

Замість того щоб оркестратор "запам'ятовував" output кожної фази в context window, записуй output у файли:

```markdown
## Inter-Phase Communication

- BA writes requirements to `docs/plans/{task}/requirements.md`
- Developer reads requirements, writes implementation summary to `docs/plans/{task}/implementation.md`
- Each QG agent reads implementation.md, writes report to `docs/plans/{task}/qg-{agent}.md`
- Orchestrator reads ONLY the QG reports (not all previous phases)
```

Це вирішує проблему context overflow: оркестратор не тримає весь history в пам'яті, а читає тільки файли поточної фази.

### 4. Enforcement через hooks замість honor system

Замість prompt-based "FORBIDDEN: Read project code", використай PreToolUse hooks:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Glob|Grep",
        "hooks": [{
          "type": "command",
          "command": "echo $CLAUDE_TOOL_INPUT | jq -r '.path // .pattern // .directory' | grep -qE '^(app|resources|database|tests|routes|config)/' && echo 'BLOCKED: Orchestrator must not read project code. Dispatch an agent instead.' && exit 1 || exit 0"
        }]
      }
    ]
  }
}
```

Це переводить "please don't read code" з honor system в **actual enforcement**. Модель фізично не зможе прочитати `app/Actions/...` без dispatch через агента.

### 5. Pipeline як slash command замість implicit instruction

Замість того щоб сподіватись що оркестратор "сам зрозуміє" pipeline, зроби explicit commands:

```
/feature "user registration with email verification"
→ Automatically: BA → checkpoint → Developer → checkpoint → QG → PR

/bugfix "category sort broken on dashboard"
→ Automatically: Debugger → Developer → Verify → Done

/review #142
→ Already exists and works well!
```

Це дає юзеру контроль, а моделі — чіткий trigger для конкретного pipeline.

## Резюме

|Аспект|Надійність|Причина|
|---|---|---|
|Перший dispatch (classify → agent)|🟢 90%|Простий, в першому повідомленні|
|Orchestrator не чіпає код|🟡 80%|Prompt-based, не enforced|
|Subagent execution|🟢 85%|Стабільна функція Claude Code|
|Context passing між фазами|🟡 60%|Оркестратор "спрощує" при передачі|
|Agent Teams (parallel QG)|🟠 50%|Experimental, 4 agents = fragile|
|Devil's Advocate interaction|🟠 40%|4+ rounds async між 3 агентами|
|Full auto pipeline end-to-end|🔴 30%|7+ steps, context drift, no enforcement|

**Головний висновок:** Твій сетап — це відмінний _design document_ для SDLC процесу. Але щоб він НАДІЙНО працював як автоматична оркестрація, потрібно:

1. Додати checkpoint'и між фазами (юзер в loop)
2. Замінити Agent Teams на sequential subagents (поки Teams не стабілізується)
3. Використати файли для передачі контексту між фазами
4. Перевести tool restrictions з prompt у hooks (enforcement)
5. Обгорнути pipeline в explicit slash commands

Це перетворить "красивий план що працює 30% часу" на "надійний pipeline що працює 85%+ часу".
