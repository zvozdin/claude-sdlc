# Експертна оцінка claude-laravel

## Загальний вердикт: 8.5/10

Олексій, це один з найпродуманіших Claude Code сетапів, які я бачив у відкритому доступі. Він не просто "набір промптів" — це повноцінна **операційна система для розробки**, побудована на правильних інженерних принципах. 110 зірок і 31 форк для нішевого конфігураційного репо — це підтвердження того, що підхід резонує з ком'юніті.

Нижче — детальний розбір по кожному аспекту з конкретними рекомендаціями.

---

## 🏆 Що зроблено відмінно

### 1. Orchestrator-as-Dispatcher паттерн (10/10)

Це найсильніша сторона всього сетапу. В `CLAUDE.md` чітко прописано:

> "You are a DISPATCHER. Your job is classification → delegation → synthesis of reports."

А в `workflow.md` — жорсткі обмеження на інструменти оркестратора:

> FORBIDDEN: Read/Grep/Glob on project code, Bash for anything beyond gh status checks, Edit/Write on any project file.

Це **принципово правильний** підхід. Більшість людей дозволяють основному агенту робити все, і він починає "трохи підглядати в код" замість того, щоб делегувати. Ти явно заборонив це, і навіть додав "маячок": "If you find yourself opening app/Actions/... — STOP."

Рішення tree: Trivial → handle directly; Bug → debugger pipeline; Feature → ba first; Ambiguous → ask ONE question — це елегантна класифікація, яка дійсно працює.

### 2. Agent Teams з паралельним Quality Gate (9/10)

Архітектура pipeline з послідовними та паралельними фазами — це вже рівень production CI/CD:

```
ba → developer ═══╗
                   ║
       ╔═══════════╩═══════════╗
       ║   Quality Gate Team    ║
       ║  tester | reviewer |   ║
       ║  security-scanner | qa ║
       ╚═══════════╤═══════════╝
                   ║
             docs-writer
```

Особливо вдале рішення — max 2 retry cycles з ескалацією до юзера. Це запобігає нескінченним loop-ам, де агенти "фіксять" знахідки одне одного.

Planning Team з devil's advocate — це взагалі рідкісний паттерн, який я майже ніде не бачив. Ідея challeng-ити requirements ДО написання коду через adversarial агента — це дуже зріле архітектурне рішення.

### 3. Principle of Least Privilege для tools (9/10)

Кожен агент має explicit `tools:` список у frontmatter. Reviewer не має Write/Edit. Devil має тільки SendMessage + Read. Docs-writer має GitHub MCP для PR, але не має security tools. Це **правильний підхід до безпеки агентів**.

Конкретні приклади того, як це працює:

- `devil` — SendMessage, Read, Glob, Grep (read-only critic)
- `reviewer` — Read, Glob, Grep, Bash + GitHub MCP для PR коментарів (read + review)
- `developer` — Read, Glob, Grep, Edit, Write, Bash + Context7 + Figma (full write access)
- `docs-writer` — все для документації + GitHub MCP для створення PR (model: haiku — правильна економія)

### 4. Розумний вибір моделей (8/10)

Модель підібрана під складність задачі:

- **opus** — ba, ddd-architect, devil, filament, frontend, laravel-refactoring-expert, qa, reviewer, security-scanner, tester (складні аналітичні задачі)
- **sonnet** — developer, dba, debugger, devops, integration-architect, queue-specialist (імплементація)
- **haiku** — docs-writer (генерація тексту)
- **opusplan** — orchestrator (головний model у settings.json)

Це показує розуміння cost/quality trade-off. Docs-writer на haiku — чудово, не треба платити за opus щоб написати README.

### 5. MCP інтеграція (8/10)

Набір MCP серверів покриває реальні потреби:

- `laravel-boost` — Laravel-специфічні інструменти прямо з Docker
- `context7` — актуальна документація фреймворків (замість галюцинацій)
- `figma` — дизайн-to-code pipeline
- `github` — повна GitHub інтеграція для PR review/creation

Агенти мають доступ тільки до тих MCP, які їм потрібні — `developer` має Figma і Context7, `reviewer` має GitHub MCP, `ba` має Context7 для дослідження.

### 6. Hooks на Stop (8/10)

```json
"Stop": [
  { "command": "docker compose exec app composer run pint:fix" },
  { "command": "docker compose exec app composer run rector:fix" }
]
```

Автоматичний code formatting і modernization при кожному завершенні сесії — це "safety net", який гарантує що навіть якщо агент пропустив стандарти, код все одно буде відформатований.

### 7. Білінгвальність (7/10)

Trigger words на EN + UA у кожному агенті, приклади на обох мовах, і "Language: Ukrainian or English based on user preference" — це продумана локалізація, яка рідко зустрічається в open-source сетапах.

## 🟡 Що можна покращити

### 1. Context Window Overload (Критично)

**Проблема:** CLAUDE.md використовує @-imports:

```
@.claude/rules/workflow.md
@.claude/rules/code-style.md
@.claude/rules/git-operations.md
```

Ці три файли завантажуються в КОЖНУ сесію. Тільки `workflow.md` — це 224 рядки. Разом з CLAUDE.md, 10 rules (які auto-loaded), і описами 17 агентів — це значна частина context window ще ДО початку роботи.

**Рекомендація:** Перевести більшість правил зі rules (always loaded) у skills (loaded on demand). Rules залишити тільки для того, що реально потрібно ЗАВЖДИ: workflow.md і git-operations.md. Code-style, architecture, testing, forms-authorization, inertia-vue, docker-commands, mcp-stack, migrations-queue — все це має бути skills з хорошими description triggers. Це заощадить 60-70% контексту в більшості сесій. Ти, до речі, вже почав це робити — в агентах є таблиці "Skills to Activate". Просто доведи цю практику до кінця.

### 2. Немає Validation/Verification між фазами (Важливо)

**Проблема:** Pipeline передбачає, що кожна фаза успішна. Але що якщо `ba` видав неповні requirements? Або `developer` зламав build? Між фазами немає explicit check-ів.

**Рекомендація:** Додай verification steps в workflow.md:

- Після `ba`: orchestrator перевіряє чи є user stories + acceptance criteria
- Після `developer`: orchestrator запускає `docker compose exec app composer run pint:fix && phpstan` ДО quality gate
- Після quality gate: orchestrator перевіряє чи всі 4 агенти дали відповідь

### 3. Відсутня Error Recovery Strategy (Важливо)

**Проблема:** Що відбувається коли:

- MCP сервер не відповідає (Docker down, Context7 timeout)?
- Агент вичерпує context window?
- GitHub MCP не може створити PR?

Зараз це не описано ніде — агенти просто зафейляться.

**Рекомендація:** Додай в кожного агента секцію "Fallback Behavior":

- Якщо MCP недоступний → використай CLI альтернативу (ти вже робиш це в review-pr.md з `gh` CLI fallback — розповсюдь цей паттерн)
- Якщо build fails → зупинись, покажи error юзеру, не передавай в quality gate
- Якщо context window наближається до ліміту → відправ проміжний звіт і запропонуй продовжити в новій сесії

### 4. Дублювання між Skills (Помірно)

**Проблема:** Є дублювання знань між деякими скілами:

- `playwright-expert` і `playwright-skill` — два Playwright скіли
- `database-optimizer`, `postgresql`, `postgres-best-practices` — три DB скіли
- `architecture-designer` і `laravel-architecture` — перетинаються
- `github-actions` має приклади для dotnet і node, хоча це Laravel-specific сетап

**Рекомендація:** Консолідувати:

- Один Playwright скіл з references
- Один PostgreSQL скіл з references (postgres-best-practices вже має хорошу структуру з rules/)
- Архітектурні скіли: один generic + один Laravel-specific

### 5. Немає Hooks для PreToolUse (Помірно)

**Проблема:** Зараз є тільки Stop hook. Але для production сетапу корисні й інші:

**Рекомендація:**

- `PreToolUse` на `Write`/`Edit` — перевіряти чи файл у дозволеній директорії (запобігає випадковим змінам .env, vendor/, node_modules/)
- `PostToolUse` на `Bash` — логувати всі виконані команди для audit trail
- `PreToolUse` на `Bash` — блокувати деструктивні команди (rm -rf, DROP TABLE, git push --force)

### 6. Settings.json — deny list занадто короткий (Помірно)

**Проблема:** Зараз denied тільки `Read(./.env)`. Але є інші чутливі файли.

**Рекомендація:** Розширити:

```json
"deny": [
  "Read(./.env)",
  "Read(./.env.local)",
  "Read(./.env.production)",
  "Read(**/.env*)",
  "Read(**/credentials*)",
  "Edit(.claude/**)",
  "Write(.claude/**)"
]
```

### 7. Немає Onboarding Flow (Незначно)

**Проблема:** Нового юзера зустрічає 17 агентів, 10 правил, 23 скіли — без поступового введення. README описує встановлення, але не описує "як почати використовувати".

**Рекомендація:** Додай command `/onboarding` або `/getting-started`, який:

1. Перевіряє чи Docker запущений
2. Перевіряє чи MCP сервери доступні
3. Пропонує простий перший таск: "Спробуй: 'Add a simple health check endpoint'"
4. Пояснює pipeline на цьому прикладі

## 🔵 Архітектурні спостереження

### Skills з References — правильний паттерн

Скіли типу `security-reviewer`, `playwright-expert`, `database-optimizer` мають піддиректорію `references/` з деталізованими файлами. Це progressive disclosure: SKILL.md завантажується завжди, а references — тільки коли потрібно. Це найкращий паттерн для управління context window, і ти його вже застосовуєш.

### review-pr Command — production-ready

Команда `/review-pr` — це, мабуть, найзріліший компонент всього сетапу. Вона:

1. Парсить різні формати PR reference (#123, repo#123, full URL)
2. Паралельно збирає metadata + diff + CI checks + existing comments
3. Делегує в reviewer агент з повним контекстом
4. Використовує 3-step GitHub MCP workflow (create pending → add inline comments → submit)
5. Має fallback на gh CLI
6. Явно забороняє emoji і згадки AI

Це рівень tooling, який я б очікував від внутрішнього інструменту серйозної команди.

### docs-writer з GitHub MCP — розумне рішення

Те, що docs-writer має доступ до `create_pull_request`, `create_branch`, `push_files` — це завершує pipeline: останній агент не тільки пише документацію, а й створює PR. Це full automation від requirements до PR.

### opusplan як orchestrator model

Використання `opusplan` для головного потоку — це правильний вибір. Orchestrator потребує planning capability для класифікації і routing, але не потребує великого context для коду.

## 📊 Порівняння з аналогами

### vs bug-ops/claude-plugins (Rust agents)

Bug-ops має схожий паттерн (team-develop, team-debug), але їхні агенти більш generic і не мають:

- explicit tools restrictions у frontmatter
- adversarial devil's advocate
- MCP integration depth (Figma, Laravel Boost, Context7)
- hooks для автоформатування

**Твій сетап глибший і більш production-ready.**

### vs laravel/agent-skills (офіційний Laravel)

Офіційний Laravel plugin — це 3 скіли (laravel, laravel-cloud, nightwatch). Твій сетап — це 17 агентів + 23 скіли з повним SDLC pipeline. Вони не конкурують — вони доповнюються. Ти можеш інтегрувати офіційні Laravel skills як додаткове джерело знань для своїх агентів.

### vs vizra-ai/claude-code-agents (59 agents)

Vizra має більше агентів (59), але вони "плоскі" — немає оркестрації, pipeline, або team collaboration. Це просто колекція промптів для копіювання. Твій сетап архітектурно набагато зріліший.

## Підсумок

Олексій, ти побудував щось, що виходить за рамки "конфігурації Claude Code". Це по суті **framework для AI-driven SDLC** — з чіткою оркестрацією, розділенням відповідальностей, security controls, і production-ready tooling. Найсильніші сторони — orchestrator-as-dispatcher паттерн, devil's advocate у planning фазі, і принцип найменших привілеїв для інструментів агентів.

Головні зони росту — context window management (переведення rules → skills), error recovery, і підготовка архітектури до multi-stack розширення, про яке ми говорили раніше.

Якщо говорити простіше: цей сетап — це Senior Engineer level. Щоб стати Staff Engineer level, йому потрібні reliability patterns (fallbacks, verification, error recovery) і extensibility (plugin architecture для multi-stack).
